# 🚀 배드민턴 플랫폼 'B-Match' 뉴스 요약 자동화 파이프라인 구축 기획서

본 프로젝트는 배드민턴 올인원 서비스 플랫폼 **'B-Match(비매치)'**의 마케팅 및 유저 정보 제공 레이어를 강화하기 위한 백엔드 데이터 자동화 파이프라인입니다. 수동 수집 및 요약 작업으로 소모되는 리소스를 최소화하고, 노션 데이터베이스와의 완벽한 연동을 통해 실시간 도메인 뉴스를 적재하는 시스템을 구축했습니다.

---

## 1. 자동화 워크플로우 아키텍처

### 1) 워크플로우 전체 구조
매일 지정된 오전에 뉴스 소스(RSS) 수집을 시작으로, 핵심 도메인 필터링과 생성형 AI(Gemini) 가공 단계를 거쳐 최종 협업 도구(Notion)에 정형 데이터로 자동 적재되는 4단계 엔드투엔드 파이프라인입니다.

![전체 워크플로우 구조]
<img width="1057" height="698" alt="image" src="https://github.com/user-attachments/assets/38e400d1-8409-4471-b569-8c8185564d6a" />


### 2) 각 단계별 역할 및 매커니즘
* **[수집 레이어] RSS (Watch RSS feed items)**
  * 특정 매체에 국한되지 않는 전방위적인 정보 확보를 위해 구글 뉴스 RSS 피드를 채택했습니다.
  * 불필요한 트래픽 유발 및 인퍼런스 비용을 억제하기 위해 **최대 반환 아이템 수(Maximum number of returned items)를 `1`개로 엄격하게 제약**했습니다.
  * **스케줄 트리거:** `Daily` 옵션을 적용하여 매일 아침 지정된 시간에 시스템 개입 없이 자동 실행되도록 설정했습니다.
* **[정제 레이어] 주제 필터링 (Data Filter)**
  * 무분별한 뉴스를 필터링하고 플랫폼 정체성에 부합하는 알짜 정보만 선별하기 위해 RSS 모듈과 Gemini 모듈 사이에 유기적인 필터 구조를 도입했습니다.
  * 기사의 `Title(제목)` 속성을 검사하여 **'배드민턴'** 키워드가 포함(Contains [case insensitive])된 데이터 객체만 통과시키는 데이터 정제 체계를 확립했습니다.
* **[가공 레이어] Google Gemini AI (Generate a response)**
  * 필터링을 통과한 핵심 기사의 제목과 링크를 LLM 프롬프트에 동적으로 주입하여, 사용자 친화적인 'AI 3줄 요약본'을 추출하는 가공 코어 엔진입니다.
* **[저장 레이어] Notion (Create a Database Item - Enter manually 모드)**
  * 데이터 파싱 서식 오류를 완전 차단하기 위해 수동 매핑 모드를 적용했습니다. 노션 데이터베이스 스키마 속성에 맞게 제목(`Title`), AI 3줄 요약(`Rich Text`), 원문 링크(`URL`), 발행 일시(`Date`) 속성을 1:1로 정밀 정렬하여 레코드를 생성합니다.

## 2. 미션 조건별 상세 세팅 내용 (평가 요소 만족 여부)

### 1) 주제 필터링 및 선택 이유
* **세팅 사양:** `RSS.Title` ➔ `Contains (case insensitive)` ➔ `배드민턴`
* **선택 이유:** 일반적인 IT/스타트업 기술 뉴스를 일방향적으로 수집하는 것보다, B-Match 플랫폼의 주 타겟층인 동호인 및 클럽 관리자들에게 즉각적으로 가치 있는 대회 소식, 신제품 트렌드, 부상 방지 정보를 실시간으로 큐레이션 공급하기 위해 도메인 키워드 필터를 구축했습니다.

### 2) 비용 및 중복 방지 정책
* **중복 저장 방지 키:** `GUID` / `URL` 기반 자동 필터링 적용
* **비용 최적화 설계:** Make RSS 모듈 내부 엔진에 내장된 고유 주소(URL) 및 고유 식별자(GUID) 캐싱 메커니즘을 적극 채택했습니다. 매일 자동 스케줄러가 트리거링되더라도, 이미 과거에 노션에 적재 완료된 뉴스 기사는 요약 API 호출 대상에서 원천 배제됩니다. 이를 통해 기사 1건당 요약 호출 1회 원칙을 완벽히 수행하며 불필요한 크레딧 소모를 0원으로 수렴시켰습니다.

### 3) 시스템 안정성 및 에러 처리 정책 (재시도 상한 2회)
* **에러 대응 메커니즘:** 생성형 AI(Gemini) 무료 티어 플랜 특성상 발생할 수 있는 일시적 트래픽 초과(`429 Rate Limit`) 및 구글 서버 과부하(`503 High Demand`) 에러에 유연하게 대응하기 위해 예외 처리 레이어를 구축했습니다.
* **재시도 정책 상한:** 시나리오 글로벌 옵션에서 `Store incomplete executions (미완료 실행 저장)` 설정을 `Yes`로 활성화한 뒤, 제미나이 모듈에 **`Retry` 에러 핸들러**를 장착했습니다. 에러가 나면 무작정 크래딧을 날리거나 멈추는 대신 구글 서버가 숨을 고를 시간인 10초 간격(`Interval`)을 두고 **최대 2회까지만 자동 재시도(`Number of attempts: 2`)**하도록 상한선을 엄격히 규정하여 시스템의 견고함과 비용 안전성을 동시에 확보했습니다.

## 3. 최종 데이터베이스 적재 결과

자동화 파이프라인 가동 후 노션 데이터베이스(`배드민턴 뉴스 자동화 저장소`)에 실시간 데이터가 속성 포맷에 맞춰 정상적으로 인서트된 결과물입니다.

![최종 노션 데이터베이스 저장 결과]
<img width="937" height="754" alt="image" src="https://github.com/user-attachments/assets/c4ddeb25-9be0-41a6-ac74-a20f6b11f29d" />


* **제목:** RSS에서 수집한 기사 제목이 노션 `Title` 타입으로 저장됨
* **AI 3줄 요약:** Gemini가 요약한 텍스트가 줄바꿈이 유지된 채 `Rich Text` 타입으로 적재됨
* **👉 원문 링크:** 뉴스 출처로 이동 가능한 하이퍼링크가 `URL` 속성 규칙에 맞게 저장됨
* **발행 일시:** 기사 업로드 타임스탬프 데이터가 노션 `Date` 포맷에 연동되어 캘린더 및 날짜 형식으로 정확히 매핑됨
